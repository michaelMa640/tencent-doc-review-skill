# 真实 MCP 能力验证记录：下载优先更新

## 1. 记录信息

- 日期：2026-03-26
- 验证环境：OpenClaw
- 验证目标：确认腾讯文档 MCP 是否支持直接下载原始 `.docx`，以及是否支持指定本地下载目录

## 2. 新增已验证能力

### 2.1 可以直接下载原始 Word

用户实测结果表明：

- 腾讯文档 MCP 可以直接把腾讯文档下载为 `.docx`
- 下载成功时返回明确的本地文件路径

这意味着：

- v3 不必再把“读取正文并本地生成 Word”当作唯一主路径
- 原始 Word 下载应当提升为优先路径

### 2.2 可以下载到指定本地目录

用户实测结果表明：

- 可以通过自然语言指定下载目录
- 例如将文档下载到：
  - `C:\Users\VBTvisitor\Desktop\腾讯文档测试`

这意味着：

- Bridge 协议必须支持“preferred_download_dir”
- skill workflow 需要显式透传下载目录

## 3. 仍需注意的现实问题

### 3.1 成功率受提示词影响

同一个下载意图在不同表述下，曾出现：

- 先误判为不能下载
- 先走到权限解释或浏览器手动下载建议
- 再次追问后才真正调用 MCP 并下载成功

结论：

- Bridge prompt 需要更硬约束
- 必须显式要求优先使用腾讯文档 MCP 下载
- 必须禁止 web search 或浏览器解释性兜底

### 3.2 仍然需要 fallback

虽然直接下载已经验证可行，但仍然不能保证所有文档、所有提示词、所有运行时都百分之百成功。

因此仍需保留：

- `直接下载失败 -> 读取正文 -> 本地生成 .docx`

## 4. 对产品路线的影响

v3 的真实优先路径更新为：

`MCP 直接下载原始 .docx 到指定本地目录 -> 本地批注 -> 上传到腾讯文档指定位置`

fallback 路径更新为：

`直接下载失败 -> 读取正文 -> 本地生成 .docx -> 本地批注 -> 上传`

## 5. 对代码结构的影响

本次验证要求代码做出以下调整：

1. bridge export 协议优先返回 `source_path`
2. `DownloadManager` 优先消费 `source_path`
3. `SkillRequest` 显式支持 `download_directory`
4. `openclaw_bridge.py` 优先提示“下载 docx 到指定目录”，失败再正文 fallback

## 6. 当前结论

腾讯文档 MCP 的可行性比此前判断更强：

- 不仅可以读取
- 还可以直接下载原始 Word
- 并且可以指定本地下载目录

因此 v3 已经从“正文物化优先”正式切换到“下载优先，正文 fallback”。
