# 真实 MCP 能力验证记录

## 1. 记录信息

- 日期：2026-03-26
- 验证环境：OpenClaw 飞书端
- 验证目标：确认腾讯文档官方 MCP 是否能支撑 v3 主链路

## 2. 已验证能力

### 2.1 能读取腾讯文档

- 已验证 OpenClaw 中的腾讯文档官方 MCP 可以访问并操作当前账号下的腾讯文档。
- 已验证能根据自然语言定位文档并读取内容。

结论：

- 可以将官方 MCP 作为 v3 的文档访问层。

### 2.2 能上传到指定位置

已验证可以通过自然语言完成以下操作：

- 把本地文档上传到腾讯文档
- 在“个人空间”中新建文件夹
- 将上传后的文档放入该文件夹
- 返回新文件链接与文件夹链接

用户实测结果表明：

- “个人空间”可作为有效上传目标
- 可以新建目标文件夹后再上传/移动文档
- 上传到指定位置是可行主路径

结论：

- v3 可以把“上传到腾讯文档指定位置”作为正式交付能力。

## 3. 未验证或受限能力

### 3.1 不能直接导出原始 Word

当前未验证到以下能力：

- 通过官方 MCP 直接把腾讯文档导出为原始 `.docx`

实际可行方式是：

- 先读取文档正文
- 再在本地生成新的 Word 文档

结论：

- v3 不应再以“下载原始 Word”作为主路径。

### 3.2 未验证在线原生批注

当前未验证到稳定可用的“在线原生批注/comment”能力。

结论：

- v3 的批注载体应放在本地生成的 Word 文档中，而不是腾讯文档在线原文中。

## 4. 空间模型验证结论

本次验证明确发现：

- “个人空间”不是“我的云盘/文档区”的同义词
- 上传目标需要显式区分空间类型和目标文件夹

因此上传协议至少需要保留：

- `space_type`
- `space_id`
- `folder_id`
- `path_hint`

## 5. 对 PRD 的影响

本次验证后，v3 主链路应统一改为：

`MCP 读取腾讯文档正文 -> 本地生成 Word -> 本地批注 -> 上传到腾讯文档指定位置`

不再使用以下表述作为主路径：

- “下载原始 Word”
- “在线原生批注写回”

## 6. 对代码结构的影响

需要调整的方向：

1. Access 层从“下载文件”转为“读取内容并物化为 Word”
2. UploadTarget 需要显式表达目标空间类型
3. Skill workflow 需要围绕“文本输入 -> Word 输出”重排

## 7. 当前结论

官方 MCP 目前已经足够支撑 v3 的 MVP，但主链路应当是：

- 读取正文
- 本地生成 Word
- 本地写批注
- 上传到指定位置

这条路径成立，且比继续依赖在线批注或直接导出 Word 更稳。
